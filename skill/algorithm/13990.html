例题：2376. 统计特殊整数

要解决这道题需要前置知识「位运算与集合论」，在这简述一下：

集合可以用二进制表示，二进制从右往左第 iii 位为 111 表示 iii 在集合中，为 000 表示 iii 不在集合中。例如集合 {0,2,3}\left\{ 0,2,3 \right\}{0,2,3} 对应的二进制数为 1101(2)1101_{(2)}1101(2)​

设集合对应的二进制数为 xxx，本题需要用到两个位运算操作：

  • 判断元素 ddd 是否在集合中：x >> d & 可以取出 xxx 的第 ddd 个比特位，如果是 111 就说明 ddd 在集合中
  • 把元素 ddd 添加到集合中：x = x|(1<<d)

思路：

将 nnn 转换为字符串 sss，定义 f(i,mask,isLimit,isNum)f(i,mask,isLimit,isNum)f(i,mask,isLimit,isNum) 表示构造第 iii 位及其之后数位的合法方案数，其余参数的含义：

  • maskmaskmask 表示当前选过的数字集合，换句话说，第 iii 位要选的数字不能再 maskmaskmask 中
  • isLimtisLimtisLimt 表示当前是否受到了 nnn 的约束（要构造的数不能超过 nnn）。若为真，则第 iii 位填入的数字至多为 s[i]s[i]s[i]，否则可以是 999，如果在受到了约束的情况下填入了 s[i]s[i]s[i]，那么后续填入的数字任然会受到 nnn 的约束。例如 n=135n=135n=135，那么 i=0i=0i=0 填的是 111 的话，i=1i=1i=1 的这一位最多填 333，否则可以填入 999
  • isNumisNumisNum 表示 iii 前面的数位是否填了数字。若为假，则当前位可以跳过（不填数字），或者填入的数字至少为 111；若为真，则要填入的数字可以从 000 开始。例如 n=123n=123n=123，在 i=0i=0i=0 时跳过的话，相当于后面要构造的是一个 999999 以内的数字了，如果 i=1i=1i=1 不跳过，那么相当于构造一个 101010 到 999999 的两位数，如果 i=1i=1i=1 跳过，相当于构造的是一个 999 以内的数

实现细节：

递归入口 f(0, 0, true, false)，表示：

  • 从 s[0]s[0]s[0] 开始枚举；
  • 一开始集合中没有数字
  • 一开始要受到 nnn 的约束（第一位必须受到约束）
  • 一开始没有填数字

递归中：

  • 如果 isNumisNumisNum 为假，说明前面没有填数字，那么当前也可以不填数字。一旦从这里递归下去，isLimitisLimitisLimit 就可以置为 falsefalsefalse 了，这是因为 s[0]s[0]s[0] 必然是大于 000 的，后面就不受到 nnn 的约束了。或者说，最高位不填数字，后面无论怎么填都比 nnn 小。
  • 如果 isNumisNumisNum 为真，那么当前必须填一个数字，枚举填入的数字，根据 isNumisNumisNum 和 isLimtisLimtisLimt 来决定填入数字的范围

递归终点：当 iii 等于 sss 的长度时，如果 isNumisNumisNum 为真，则表示得到了一个合法数字（因为不合法的不会继续递归下去），返回 111 ，否则返回 000。

答疑

问：isNumisNumisNum 这个参数是否可以去掉？

答： 由于 markmarkmark 中记录了数字，可以通过判断 markmarkmark 是否为 000 来判断前面是否填了数字，所以 isNumisNumisNum 可以省略。

下面的代码保留了 isNumisNumisNum，主要是为了方便掌握模版。因为有些题目不需要 markmarkmark，但需要 isNumisNumisNum。

**问：**记忆化四个状态有点麻烦，能不能只记忆化 (i,mark)(i,mark)(i,mark) 这两个状态？

答： 是可以的。比如 n=234n=234n=234，第一位填 222，第二位填 333，后面无论怎么递归，都不会再递归到第一位填 222，第二位填 333 的情况，所以不需要记录。又比如，第一位不填，第二位也不填，后面无论怎么递归也不会再次递归到这种情况，所以也不需要记录。

根据这个例子，我们可以只记录不受到 isLimitisLimitisLimit 或 isNumisNumisNum 约束的状态 (i,mark)(i,mark)(i,mark)。比如 n=234n=234n=234，第一位（最高位）填的 111，那么继续递归，后面就可以随便填，所以状态 (1,2)(1,2)(1,2) 就表示前面填了一个 111 （对应的 mark=2mark =2mark=2），从第二位往后随便填的方案数。

**问：**能不能只记忆化 iii ？

答： 这是不行的。想一想，我们为什么要用记忆化？如果递归到同一个状态时，计算出的结果就可能是不一样的。如果只记忆化 iii，就可能会算出错误的结果。

也可以这样理解：记忆化搜索要求递归函数无副作用（除了修改 memo 数组），从而保证递归到同一个状态时，计算出的结果是一样的。

902. 最大为 N 的数字组合

因为这道题填入的数字是直接从 digitsdigitsdigits 中选择，所以可以胜利第二个参数，然后只记忆化 iii 就可以了，因为：

  1. 对于一个固定的 iii，它受到 isLimitisLimitisLimit 或 isNumisNumisNum 的约束在整个递归过程中至多会出现一次，没必要记忆化。比如 n=234n=234n=234，当 i=2i=2i=2 的时候，前面可以填 11,12,13,...,2311,12,13,...,2311,12,13,...,23，如果受到 isLimtisLimtisLimt 的约束，就说明前面填的是 232323。「当 i=2i=2i=2 的时候，前面填的是 232323」这件事情，在整个递归过程中至多会出现一次。
  2. 另外，如果只记忆化 iii，dpdpdp 数组的含义就变成在不受到 nnn 的约束时的合法方案数，所以要在 !isLimit && isNum 成立时才去记忆化。接着上面的例子，在前面填 232323 的时候，下一位填的数字不能超过 444，因此算出来的结果是不能套用到前面甜的是 11,12,13,...11,12,13,...11,12,13,... 这些数字上面的。

1067. 范围内的数字计数

1397. 找到所有好字符串

依旧是套用上面的模版，不同的是在递归时，进入递归的时机，由kmp算法提供，注意几个小细节即可：

  • 由于 nextnextnext 数组中并没有 −1-1−1，进入下一次递归时，当 kmpkmpkmp 指针在字符串头部时，也会出现 d≠evil[nxt]d \neq evil[nxt]d=evil[nxt] 的情况，若直接将 nxt+1nxt+1nxt+1 进入递归，是认为两个字符串匹配上了，实际则可能不是，这里需要将 nxtnxtnxt 减 111 后进入递归。使用多一位的 next 数组也可以，将循环条件改为 for nxt > -1 && d != evil[nxt] 即可。
