前言

本文将扫清位运算的迷雾，在集合论与位运算之间建立一座桥梁。

在高中，我们学了集合论（set theory）的相关知识。例如，包含若干帧数的集合 S={0,2,3}S=\left\{ 0,2,3 \right\}S={0,2,3}。在编程中，通常用哈希表（hash table）实现集合，例如 Java 中的 HashSet，Golang 中的 map。

在集合论中，有交集 $ \cap$、并集 ∪\cup∪、包含于 ⊆\subseteq⊆ 等等概念。如何编程实现「求两个哈希表的交集」，需要一个个遍地遍历哈希表中的元素。有没有更高效的做法呢？

该二进制上场了。

集合可以用二进制表示，二进制从低到高第 iii 位为 1 表示 iii 在集合中，为 0 表示 iii 不在集合中。例如集合 {0,2,3}\left\{ 0,2,3 \right\}{0,2,3} 可以用二进制数 1101 表示；反过来，二进制数 1101 就对应集合 {0,2,3}\left\{ 0,2,3 \right\}{0,2,3}。

正式地说，包含非负整数的集合 SSS 可以用如下方式「压缩」成一个数字：

f(S)=∑i∈S2if(S) =\sum_{i\in S} 2^i f(S)=i∈S∑​2i

上面举例的 0,2,3{0,2,3}0,2,3 可以压缩成 20+22+23=132^0+2^2+2^3=1320+22+23=13，也就是二进制数 1101。

利用位运算「并行运算」的特点，我们可以高效地做一些和集合相关的运算，按照常见的应用场景，可以分为以下四类：

  1. 集合与集合
  2. 集合与元素
  3. 遍历集合
  4. 枚举集合

为什么要写成 sub = (sub - 1) &s 呢？

暴力做法是从 sss 出发，不断减一直到 0，但这样中途会遇到很多并不是 sss 的子集的情况。例如 s=10101s=10101s=10101 时，减一得到 101001010010100，这是 sss 的子集。但再减一就得到 100111001110011 了，这并不是 sss 的子集，下一个子集应该是 100011000110001。

把所有的合法子集按顺序列出来，会发现我们做的相当于「压缩版」的二进制减法，例如

10101→10100→10001→10000→00101→...10101 \rightarrow 10100 \rightarrow 10001 \rightarrow 10000 \rightarrow 00101 \rightarrow ... 10101→10100→10001→10000→00101→...

如果忽略掉 101011010110101 中的两个 000，数字的变化和二进制减法是一样的，即

111→110→101→100→011→...111 \rightarrow 110 \rightarrow 101 \rightarrow 100 \rightarrow 011 \rightarrow ... 111→110→101→100→011→...

如何快速找到下一个子集呢？以 10100→1000110100 \rightarrow 1000110100→10001 为例说明，普通的二进制减法会把最低位的 111 变成 000，同时 111 右边的 000 全部变成 111，即 10100→1001110100 \rightarrow 1001110100→10011。「压缩版」的二进制减法也是类似的，把最低位的 111 变成 000，但同时对于 111 右边的 000，只保留在 s=10101s = 10101s=10101 中的 111，所以是 10100→1000110100 \rightarrow 1000110100→10001。怎么保留？ &10101\& 10101&10101 就行

但同时对于 111 右边的 000，只保留在 s=10101s = 10101s=10101 中的 111

意思是，把 101001010010100 的最低位 111 变 000，它的后面有两位 000000，都是 000，这时候按照普通二进制，会把这两个 000000 变成 111111，如果按照压缩版，就是把原来集合中有的 111 变成 111（因为求的是子集），其余的还是 000，原有的集合是101011010110101，最后两位是 010101，所以只保留 010101。综合起来就是 101001010010100 先变成 100001000010000，然后保留 010101，变成 100011000110001（按照作者的说法，应该是对于 101011010110101，只有最后一位存在值，所以最后一位取反，0→10 \rightarrow 10→1，所以 101001010010100 先变成 100001000010000，然后最后一位取反）。

这样做的效果是从 101001010010100 直接跳到了 100011000110001，把中间的 100111001110011 和 100101001010010 忽略掉了（普通减法的顺序是：10100→10011→10010→1000110100 \rightarrow 10011 \rightarrow 10010 \rightarrow 1000110100→10011→10010→10001），因为 100111001110011 和 100101001010010 不是有效子集
